## 摘要
根因分析（RCA）是对LLM智能体能力（如长上下文理解、多步推理和工具使用）的整体考验。然而，现有数据集存在一个根本性缺陷：它们仅标注根因，而不标注连接根因与观察到的症状的传播路径，这极大地简化了任务，使其退化为朴素的模式匹配。为支持严谨评估，我们引入PAVE，一种利用故障注入已知干预来重建因果传播路径的逐步标注协议。其机制是前向验证：从原因到结果进行推理，而非从症状向后推断。应用PAVE生成了OpenRCA 2.0（500个实例），这是首个为LLM智能体提供逐步因果标注的跨系统RCA基准。在11个前沿LLM中，精确恢复根因集合的平均成功率仅为20.7%。为定位难点所在，我们放宽标准并发现所谓的无依据诊断：智能体在76.0%的案例中识别出至少一个正确的根因服务，但仅61.5%的情况下将该服务与观察到的症状通过验证的因果传播路径联系起来。仅基于结果的评估隐藏了这种失败模式；逐步因果真值是构建可信的基于LLM的RCA智能体所缺失的关键环节。

## Abstract
Root cause analysis (RCA) poses a holistic test of LLM agentic capabilities, such as long-context understanding, multi-step reasoning, and tool use. However, existing datasets suffer from a fundamental gap: they label only the root cause, not the propagation path connecting it to the observed symptom, which largely simplifies the task to naive pattern matching. To support rigorous evaluation, we introduce PAVE, a step-wise labeling protocol that leverages known interventions from fault injection to reconstruct causal propagation paths. The mechanism is forward verification: reasoning from cause to effect rather than inferring backward from symptoms. Applying PAVE yields OpenRCA 2.0 (500 instances), the first cross-system RCA benchmark with step-wise causal annotations for LLM agents. Across 11 frontier LLMs, recovering the exact root-cause set succeeds in only 20.7% of cases on average. To locate where this difficulty lies, we relax the criterion and find what we call the ungrounded diagnosis: agents identify at least one correct root-cause service in 76.0% of cases, but ground that service in a verified causal propagation path to the observed symptom in only 61.5%. Outcome-only evaluation hides this failure mode; step-wise causal ground truth is the missing piece for trustworthy LLM-based RCA agents.

- **核心动机**：构建首个**步骤级因果标注**的RCA基准，推动评测从“正确局部化”转向“正确推理”。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：利用故障注入实验中的**已知干预**（`do(vroot)`），将代理面临的病态反向推断问题转化为**前向验证**问题：给定原因，检查哪些下游效应确实发生了。这种信息不对称（代理看不到干预，标注者看到）是PAVE协议的基础。
- **PAVE协议（Path Annotation via Verified Effects）**：两阶段管道，对每条候选路径施加三个必要条件（结构、统计、时序）的联合验证。

### 阶段1：结构剪枝
- 将原始遥测（时序指标、跟踪、日志）通过节点类型感知的映射`Ψ`投影到离散状态字母表`Σ`（如`HighLatency`, `CPU_Throttled`）。
- 利用传播规则集`R`和系统依赖图`G`，在时间扩展异质图上执行受限深度优先搜索，枚举所有拓扑上可能的候选路径`Π_cand`。规则基于已知机制（如RPC超时级联、资源争用）。

### 阶段2：因果验证
- 对每条候选路径，使用注入前基线窗口`D_base`构建参考分布，对每个节点执行分布适配的异常检测（Z分数/百分位数/自适应延迟检测器）。
- 路径被验证需同时满足：① 每个节点超过基线阈值；② 每条边符合规则`R`；③ 下游异常时间不早于上游原因（时序对齐）。
- 仅通过联合屏的路径进入最终验证路径集`Π*`，形成逐步因果真值图`G*`。

- **关键公式**（信息不对称）：`{O_obs} ⊂ {O_obs, do(vroot)}`。代理仅见观测，PAVE多知干预。

## 6. 主要结论与发现

1. **结果评测低估失败**：任意服务级命中率（AnySvc）平均76.0%，但最宽松的过程指标路径可达性（PR）仅61.5%，约14.5%的案例出现“无依据诊断”——代理说对根因位置，但无法构建可验证的因果路径。
2. **边F1普遍低于节点F1**：平均相差18.8个百分点（62.2% vs 43.4%），说明代理更容易识别参与服务，但难以组织正确的有向依赖关系。
3. **无单一最优模型**：不同指标的最高分分散在4个不同模型上，具体选型取决于下游验证成本。
4. **三类典型失败模式**：
   - 过早承诺（发现一个根因后停止探索）
   - 存在性偏差（将“无信号”视为正常，忽略被静默杀死的服务）
   - 显著性捕获（聚焦噪声最大的信号，忽略真实根因）
